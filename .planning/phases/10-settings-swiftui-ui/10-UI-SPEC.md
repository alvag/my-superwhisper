## Spacing Scale

SwiftUI Form `.formStyle(.grouped)` uses system-managed insets. The following tokens map to SwiftUI spacing values used within the Form content.

| Token | pt Value | SwiftUI Usage |
|-------|----------|---------------|
| xs | 4 | Icon-to-label gap within HStack rows |
| sm | 8 | Spacing between items in an HStack (vocabulary row) |
| md | 16 | Default SwiftUI Form row inset (system-managed) |
| lg | 24 | Section-to-section gap (system-managed by `.grouped`) |
| xl | 32 | Minimum window width floor (partial — see Frame below) |

**Frame contract:**
- `minWidth: 420` — only minimum width. No `minHeight`, no `maxWidth`, no `maxHeight`.
- Auto-sizing: `NSHostingController` calculates `fittingSize` from SwiftUI layout.
- Source: CONTEXT.md D-10, RESEARCH.md Pitfall 3.

Exceptions:
- Vocabulary `HStack` rows: use `.frame(maxWidth: .infinity)` if GroupedFormStyle truncates at ~600pt on macOS 26 (validate at runtime — RESEARCH.md Pitfall 4).

## Typography

SwiftUI Form `.formStyle(.grouped)` uses system text styles. All text must use system text styles — no custom font sizes.

| Role | SwiftUI Text Style | Usage | Notes |
|------|--------------------|-------|-------|
| Body | `.body` (17pt / SF Pro Regular) | Row labels, picker items, toggle labels | System default inside Form rows |
| Label | `.callout` (16pt / SF Pro Regular) | Secondary row text, TextField placeholders | Used for "Incorrecto" / "Correcto" placeholders |
| Section Header | `.headline` (17pt / SF Pro Semibold) via `Label` | Section header text | Rendered by SwiftUI inside `Section header:` |
| Button | `.body` (17pt / SF Pro Regular) | "Configurar clave API...", "Agregar corrección" | Inherits form body size |

**Line heights:** System-managed by SwiftUI. No overrides.

**Weight contract:** Two weights only:
- Regular (400) — all body text, labels, button titles
- Semibold (600) — section headers only (via `.headline` text style)

## Color

SwiftUI semantic colors only. No hex values — the system resolves light/dark mode automatically.

| Role | SwiftUI Color | Usage |
|------|---------------|-------|
| Dominant (60%) | `.windowBackground` / Form background | Main window surface — system-managed by `.formStyle(.grouped)` |
| Secondary (30%) | `.controlBackground` | Form row cells background — system-managed |
| Accent (10%) | `.accentColor` (system blue) | Reserved: Toggle thumb when ON, Picker selection highlight, keyboard focus ring |
| Destructive | `.red` (system red) | Reserved exclusively for: `minus.circle.fill` icon in vocabulary delete buttons |
| Secondary text | `.secondary` | `arrow.right` separator icon in vocabulary rows |

**Accent reserved for:**
1. Toggle control thumb when `isOn = true`
2. Picker selection highlight
3. Keyboard focus ring on TextFields and Recorder

**Destructive reserved for:**
1. `minus.circle.fill` icon on vocabulary entry delete buttons — `.foregroundColor(.red)` with `.buttonStyle(.plain)`
2. No other use of red in this phase

**Color approach note:** This is a macOS native SwiftUI app. Web hex tokens (e.g. `#FFFFFF`) are not applicable. All colors are semantic SwiftUI system colors that adapt to light/dark mode and macOS accent color preferences automatically.

**Empty state (vocabulary list):**
- Heading: none — the list is empty, and the "Agregar corrección" button is always visible below
- Body: no explicit empty state message needed — the [+] button serves as the affordance
- Rationale: System Settings style does not use empty state copy for empty lists; the add button is sufficient

**Error states:**
- Vocabulary entries with blank fields: silently ignored in pipeline (CONTEXT.md D-08). No error message shown. User clears them manually with [-].
- Launch at login failure: no UI error shown. The toggle reverts to actual SMAppService state (handled in SettingsViewModel didSet).
- API key: no validation in this phase (D-03 — no status indicator).

**Destructive actions:**
- Delete vocabulary entry: direct delete, no confirmation dialog. CONTEXT.md D-07 specifies explicit [-] button. The icon-only `minus.circle.fill` button is the affordance. No secondary confirmation — consistent with System Settings list editing pattern.

## Interaction Contract

### Vocabulary List Interactions

| Action | Trigger | Result |
|--------|---------|--------|
| Add entry | Tap "Agregar corrección" button | New row with empty TextFields appended to bottom of list |
| Edit wrong | Tap "Incorrecto" TextField | Inline edit in place, persists via `vocabularyEntries` didSet |
| Edit correct | Tap "Correcto" TextField | Inline edit in place, persists via `vocabularyEntries` didSet |
| Delete entry | Tap `minus.circle.fill` button on row | Row removed immediately, no confirmation |
| Empty field behavior | Leave TextField blank | Entry remains in list, ignored silently in pipeline (D-08) |

### Picker Interaction

| Action | Trigger | Result |
|--------|---------|--------|
| Select device | Tap device name in Picker | `selectedMicID` updated, delegates to `microphoneService.selectedDeviceID` |
| Select default | Tap "Predeterminado del sistema" | `selectedMicID` set to nil, system default mic used |

### Toggle Interactions

All three toggles: immediate persistence via `didSet + UserDefaults.standard.set`. No save/apply button.

### API Key Button

Single tap on "Configurar clave API..." → calls `viewModel.openAPIKey()` → `APIKeyWindowController` NSPanel appears as modal. No state change in SettingsView.
